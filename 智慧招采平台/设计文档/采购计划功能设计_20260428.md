## 一、功能概述

采购计划模块用于对集团的采购计划进行制定和管理，为企业提供及时准确的采购计划和执行路线。

本模块支持企业建立年度采购计划、变更子计划及采购项目立项；支持二级单位审核控股投资企业年度采购计划及分派集中采购计划。

### 模块划分

```
采购计划管理
│
├── 1. 本企业采购计划（编制端）
│   ├── 年度计划：年度计划 → 子计划（自行/集采）→ 明细
│   └── 项目计划：项目计划（仅自行采购）→ 明细
│
├── 2. 下属企业采购计划（查看端）
│   └── 本单位查看下级单位已生效的计划（年度计划/项目计划）
│
├── 3. 集采计划分派（分派端）
│   └── 二级单位将下级单位的集采子计划分派给执行单位
│
├── 4. 本企业可执行计划（执行端）
│   ├── 本企业审核通过的子计划/项目计划
│   ├── 二级单位分派的集采子计划
│   └── 支持变更明细，重新提交审核
│
├── 5. 电煤计划管理（独立页面）
│   └── 电煤类计划独立管理（因业务特殊性，详见后续专项设计文档）
│
└── 6. 采购项目立项（立项端）
    └── 基于可执行计划中的子计划/项目计划进行立项
    └── 选择至明细层级
    └── 拟采购方式自动带入，可编辑

审核入口：采购系统 → 任务中心（统一处理所有计划审核）
```

## 二、核心业务流程

采购计划模块围绕"编制 → 审核 → 执行 → 立项"的完整业务链路展开，支持年度计划和项目计划两种管理模式，并涵盖集中采购计划的特殊分派场景。

### 2.1 编制流程

```
年度计划：
  创建年度计划主表 → 添加子计划 → 添加明细 → 提交审核

项目计划：
  创建项目计划主表 → 添加明细 → 提交审核
```

### 2.2 审核流程（在任务中心处理）

```
年度计划：
  本单位审核 → 逐级审核 → 二级单位审核 → 生效
  审核不通过 → 返回编制端，状态变为审核不通过，可编辑后重新提交

子计划明细：
  内部审核 → 生效
  审核不通过 → 返回编制端，状态变为审核不通过，可编辑后重新提交

项目计划：
  内部审核 → 生效
  审核不通过 → 返回编制端，状态变为审核不通过，可编辑后重新提交
```

### 2.3 分派流程（仅集采）

```
二级单位查看已生效集采子计划 → 选择执行单位 → 分派
```

### 2.4 执行流程

```
查看可执行计划 → 变更明细（需本单位审核）→ 立项
```

### 2.5 立项流程

```
选择可执行计划中的子计划/项目计划 → 选择明细 → 创建立项

约束：
  - 一条明细仅用一次
  - 所选明细的项目类型须一致
  - 拟采购方式自动带入，可编辑
```

## 三、数据结构

### 3.1 年度计划

#### 3.1.1 年度计划结构

```
年度计划主表（1）
  └── 子计划（N）一对多关系

- 1个年度计划 → 多个子计划

  已生效 → 点击变更 → 已变更 → 编辑主表/子计划 → 提交审核 → 待审核 → 审核通过 → 已生效
                                                          ↘ 审核不通过

**子计划状态**：

| 状态    | 操作           | 说明                               |

#### 年度计划与子计划状态关系说明


**子计划状态流转图**：

```
首次编制（随年度计划）：
  草稿 → 随年度计划提交审核 → 待审核 → 审核通过 → 已生效 → 立项使用 → 已使用
                                        ↘ 审核不通过 → 审核不通过

年度计划已生效后新增子计划：
  草稿 → 随年度计划变更提交审核 → 待审核 → 审核通过 → 已生效
                                            ↘ 审核不通过 → 审核不通过

已生效子计划独立变更明细：
  已生效 → 点击变更明细 → 已变更 → 编辑明细 → 提交审核 → 待审核 → 审核通过 → 已生效
                                                                    ↘ 审核不通过 → 审核不通过
```


**核心规则**：

- 子计划状态独立管理，不随年度计划状态自动联动
- 随年度计划送审的子计划：年度计划审核结果会同步给这些子计划
  - 通过 → 子计划变为已生效
  - 不通过 → 子计划回退（新增的变草稿，基础信息变更的变已变更）
- 独立审批的子计划：不受年度计划审核结果影响
  - 明细变更走内部审核流，年度计划审核结果不影响它
- 未改动的子计划：状态保持不变
  - 子计划被分派后，状态不变，仍为"已生效"
  - 子计划被立项使用后，状态变为"已使用"

**年度计划审核完成后子计划状态变化**：

| 子计划变更前状态    | 是否参与本次变更 | 年度计划提交后     | 审核通过后   | 审核不通过后  |
| ----------- | -------- | ----------- | ------- | ------- |
| 草稿（含首次编制和年度计划变更时新增） | ✅ 是      | 待审核         | 已生效     | 审核不通过   |
| 已变更（基础信息变更） | ✅ 是      | 待审核         | 已生效     | 审核不通过   |
| 已变更（明细变更中）  | ❌ 互斥     | 不允许发起年度计划变更 | -       | -       |
| 已生效（未改动）    | ❌ 否      | 已生效（不变）     | 已生效（不变） | 已生效（不变） |
| 已使用（未改动）    | ❌ 否      | 已使用（不变）     | 已使用（不变） | 已使用（不变） |

**互斥规则**：

- 任一子计划处于"已变更（明细变更中）"状态 → 年度计划不允许发起变更
- 年度计划处于"待审核"状态 → 子计划不允许发起明细变更

| 状态    | 操作         | 说明                |

```

  已生效 → 点击变更 → 已变更 → 提交审核 → 待审核 → 审核通过 → 已生效

  已生效 → 被后续采购项目关联使用 → 已建项
```

### 5.5 采购项目立项

**功能路径**：`采购系统 → 采购立项 → 采购项目立项`

**功能定位**：基于可执行计划中的子计划/项目计划，拆分到明细级别进行立项。

**核心规则**：
- 前置条件：必须关联可执行计划中的子计划/项目计划
- 关联层级：关联到明细级别，只要子计划/项目计划下还有未使用的明细，该计划就可选
- 明细选择：选择子计划/项目计划后，必须在其内部选择具体明细
- 项目类型一致性：同一立项中，所选明细的项目类型必须一致（提交时校验）
- 采购方式：拟采购方式自动带入（第一个子计划的拟采购方式），可编辑，以立项为准
- 立项后标记：明细被立项使用后，标记为"已使用"，不可再次用于立项
- 估算金额：取已选明细的估算金额之和
- 变更规则：已生效的立项可变更，变更后需重新提交审核
- 撤回规则：待审核状态下可撤回，撤回后状态变为草稿
- 删除规则：草稿和审核不通过状态下可删除

#### 变更操作

**交互流程**：

```
立项列表 → 点击"变更" → 状态变为已变更 →

**变更页面**：

- 支持修改：关联明细（新增/删除）、立项信息、协同信息、委托单

- **关键规则**：已生效的立项可变更，变更后需重新提交审核；待审核状态下可撤回，撤回后状态变为草稿

### 7.1 审核规则

| 规则 | 说明 |

| 审核入口 | 统一在"采购系统 → 任务中心"处理 |

| 自行采购计划审核路径 | 内部审核 |

### 7.2 业务类型定义

在审批流程系统的标准业务类型库中新增：

| 业务类型编码 | 业务名称 | 适用场景 |
|------------|---------|---------|
| `PLAN_ANNUAL_APPROVAL` | 年度计划审核 | 年度计划提交审核 |
| `PLAN_PROJECT_APPROVAL` | 项目计划审核 | 项目计划提交审核 |
| `PLAN_ITEM_CHANGE` | 计划明细变更 | 已生效计划的明细变更 |
| `PROJECT_INITIATION_APPROVAL` | 采购项目立项审核 | 采购项目立项提交审核 |
| `PROJECT_INITIATION_CHANGE` | 采购项目立项变更 | 已生效立项的变更 |

### 7.3 审批路径配置

| 业务类型 | 推荐路径 | 说明 |
|---------|---------|------|
| 年度计划审核 | 本级 → 逐级 → 二级 | 使用多级审批流，根据企业终审权限配置 |
| 项目计划审核 | 本级 | 内部审批 |
| 计划明细变更 | 本级 | 内部审批 |
| 采购项目立项审核 | 本级 → 二级 | 内部审批 |
| 采购项目立项变更 | 本级 → 二级 | 内部审批 |

### 7.4 审批实例化时机

| 业务动作 | 触发时机 | 审批实例关联字段 |
|---------|---------|----------------|
| 年度计划提交 | 点击"提交审核" | `plan_annual.approval_instance_no` |
| 项目计划提交 | 点击"提交审核" | `plan_project.approval_instance_no` |
| 明细变更提交 | 变更明细后提交审核 | 新增变更记录表关联 |
| 立项提交 | 点击"提交审核" | `project_initiation.approval_instance_no` |
| 立项变更提交 | 变更立项后提交审核 | `project_initiation.approval_instance_no` |

### 7.5 审批回调处理

审批流完成后，系统需回调更新业务表状态：

**审批通过**：
- 首次提交：状态变为"已生效"，可编辑锁定解除
- 明细变更：状态变为"已生效"，更新明细数据，标记为已生效
- 立项首次提交：状态变为"已生效"，关联明细标记为"已使用"
- 立项变更：状态变为"已生效"，更新立项数据

**审批不通过**：
- 首次提交：状态变为"审核不通过"，可编辑锁定解除，记录驳回原因
- 明细变更：状态变为"已变更"，变更失效，恢复原明细，可继续编辑
- 立项首次提交：状态变为"审核不通过"，可编辑或删除
- 立项变更：状态变为"审核不通过"，变更失效，可继续编辑
