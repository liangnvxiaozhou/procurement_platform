## 一、功能概述

### 1.1 执行阶段定位

项目执行是采购项目的核心实施阶段。当采购项目在「我的项目」中确认（状态=已确认）后，项目经理进入项目执行阶段，按照采购方式对应业务流程依次推进各环节，直至完成采购结果并进入标后/成交后管理。

```
采购计划 → 采购立项 → 项目立项（我的项目）→ 【项目执行】 → 采购完成
                                        ↑
                                   已确认状态触发执行
```

### 1.2 与前序模块的衔接

| 前序模块 | 衔接点 | 说明 |
|---------|--------|------|
| 采购立项 | 项目类型、采购方式、估算金额 | 立项审批通过后可用于建项 |
| 项目立项 | 项目确认（已确认状态） | 项目确认后进入执行阶段 |
| 采购委托 | 委托单已建项 | 委托采购场景下，代理项目经理执行 |

### 1.3 执行阶段模块划分

```
项目执行
│
├── 1. 发布管理（公告/邀请函）    ← 项目经理：编制/提交审核/发布
├── 2. 采购文件管理               ← 项目经理：编制招标文件/采购文件
├── 3. 标前准备（含报名管理）     ← 项目经理：答疑/报名审核/场地安排
├── 4. 开标/开启                  ← 项目经理：开标/开启
├── 5. 评标/评审                  ← 评标委员会：评审打分
├── 6. 定标/成交                  ← 项目经理：确定中标/成交人
└── 7. 标后/成交后                ← 项目经理：发通知书/合同/归档
```

执行入口：采购系统 → 项目管理 → 我的项目 → 进入项目

## 三、通用模块抽象与差异说明

### 3.1 模块抽象总览

将九种采购方式执行流程抽象为 7 个通用模块，按执行顺序排列：

| 模块  | 通用名称         | 涵盖的采购方式差异                     | 设计状态          |
| --- | ------------ | ----------------------------- | ------------- |
| 模块1 | 发布管理（公告/邀请函） | 公开=公告，邀请=邀请函，直接采购=事前公示+邀请函    | ✅ 公告完成，邀请函待补充 |
| 模块2 | 采购文件管理       | 招标类=招标文件，非招标类=采购文件，竞价=合并到准备阶段 | 待设计           |
| 模块3 | 标前准备（含报名管理）  | 通用；竞价为"竞价准备"                  | 待设计           |
| 模块4 | 开标/开启        | 招标类=开标，非招标类=开启，竞价=竞价（实时报价）    | 待设计           |
| 模块5 | 评标/评审        | 招标类=评标，非招标类=评审，竞价=无           | 待设计           |
| 模块6 | 定标/成交        | 招标类=定标，非招标类/竞价=成交             | 待设计           |
| 模块7 | 标后/成交后       | 招标类=标后，非招标类/竞价=成交后            | 待设计           |

### 3.2 模块设计原则

- **通用优先**：先抽象通用流程和字段，再处理各采购方式的差异化需求
- **按采购方式联动**：项目关联的采购方式决定执行流程的节点编排和字段展示
- **分段推进**：每个模块独立设计，模块间通过项目状态流转衔接
- **版本可追溯**：关键环节支持版本记录和操作日志

## 四、模块设计

### 4.1 发布管理（公告/邀请函）

#### 4.1.1 功能定位

项目确认后，项目经理编制公告/邀请函，经审批通过并在到达发布时间后对外发布。公告为项目级，一个采购项目对应一个公告。

**适用场景**：

| 发布类型 | 适用采购方式 | 对象范围 |
|---------|------------|---------|
| 招标公告 | 公开招标 | 面向所有供应商 |
| 询比公告 | 询比采购（公开） | 面向所有供应商 |
| 谈判公告 | 谈判采购（公开） | 面向所有供应商 |
| 竞价公告 | 竞价（公开） | 面向所有供应商 |
| 邀请函 | 邀请招标、询比（邀请）、谈判（邀请）、竞价（邀请） | 面向指定供应商 |
| 事前公示 | 直接采购 | 面向所有供应商（公示） |

> 本期先设计**公开类公告**（招标公告、询比公告、谈判公告、竞价公告），邀请函和事前公示后续补充。

---

#### 4.1.2 公告类型横向对比

四种公开类公告的页面结构一致（6 大区域），核心差异集中在**标段公告信息的时间/竞价字段**和**其他相关说明的模板文本**。

**公告类型自动判定规则**：

| 采购方式 | 询比方式/谈判方式/竞价方式 | 公告类型 | 供应商称呼 |
|---------|------------------------|---------|----------|
| 公开招标 | — | 招标公告 | 投标人 |
| 询比采购 | 公开 | 询比公告 | 供应商 |
| 谈判采购 | 公开 | 谈判公告 | 供应商 |
| 竞价 | 公开 | 竞价公告 | 供应商 |

> 注：采购方式为邀请类时，发布类型为"邀请函"（后续补充）。

**差异概览**（详见 §4.1.3 字段定义中的个性化标注）：

| 差异维度 | 公开招标 | 询比/谈判（公开） | 竞价（公开） |
|---------|---------|----------------|------------|
| 公告发布媒体 | 采购官网 + 中国招投标公共服务平台 | 仅采购官网 | 仅采购官网 |
| 标段公告-文件获取截止默认 | 公告开始时间+5天 | +3天 | 无此字段 |
| 标段公告-澄清截止默认 | +6天 | +3天 | 无此字段 |
| 标段公告-截标/开标默认 | +21天 | +5天 | 无此字段 |
| 标段公告-竞价参数 | 无 | 无 | 竞价开始时间/类型/延时/时长/起拍价/价格梯度 |
| 其他说明-标书款支付 | ✅ 有 | ❌ 无 | ❌ 无 |
| 其他说明-平台使用费 | ❌ 无 | ✅ 有 | ✅ 有 |
| 其他说明-CA办理 | 必须办理 | 可不办理 | 可不办理 |
| 模板文本称呼 | 投标人/招标文件/招标活动 | 供应商/采购文件/采购活动 | 同询比 |

---

#### 4.1.3 数据结构

公告页面分为 6 大区域：①项目信息 ②标段/包信息 ③公告信息 ④标段公告信息 ⑤采购单位信息 ⑥其他相关说明。

- **①②③⑤⑥** 为公告主表（项目级）字段
- **④** 为标段公告子表（标段包级），一个标段包对应一条记录
- 模板文本为系统内置，按采购方式自动填充，用户可编辑覆盖（开发可考虑支持租户可配置）

##### 4.1.3.4 补遗公告表

补遗/澄清公告为独立公告记录，关联原公告，独立审批和发布。补遗公告字段结构与原公告主表一致（复用区域①②③⑤⑥），额外增加关联字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 关联原公告ID | 外键 | 系统 | 关联原公告主表 |
| 补遗类型 | 文本 | 系统 | 澄清 / 补遗，默认补遗 |
| 补遗序号 | 数字 | 系统 | 该原公告下第几次补遗（1, 2, 3...） |

**补遗公告与原公告的区别**：
- 补遗公告无需区域④标段公告信息（补遗为项目级，不按标段区分）
- 补遗公告标题默认「原公告名称+补遗公告（第N次）」，可编辑
- 补遗公告的发布时间、附件等字段同原公告逻辑
- 补遗公告拥有独立的状态流转和审批流程

**设计说明**：
- 一条原公告可发布多次补遗，补遗序号自增。
- 补遗公告拥有独立的状态流转（草稿→待审核→已发布），与原公告状态互不影响。
- 补遗公告的审批流程与原公告一致。
- 补遗公告不支持再变更和再补遗。

#### 4.1.4 公告状态字典

##### 4.1.4.1 公告状态（主表）

| 状态 | 状态码 | 说明 | 允许操作 |
|------|--------|------|---------|
| 草稿 | `DRAFT` | 编制中 | 编辑、提交审核、删除 |
| 待审核 | `PENDING_APPROVAL` | 已提交，审核中 | 查看、撤回 |
| 审核不通过 | `APPROVAL_REJECTED` | 审核拒绝 | 编辑、提交审核、删除 |
| 待发布 | `APPROVED` | 审批通过，未到发布时间 | 变更（覆盖+版本记录）、查看 |
| 已发布 | `PUBLISHED` | 到达发布时间，对外可见 | 查看、发布补遗公告、终止 |
| 已终止 | `TERMINATED` | 公告终止 | 查看 |

**状态流转图**：

```
首次编制：
  草稿 → 提交审核 → 待审核 → 审核通过 → 待发布 → 到达发布时间 → 已发布
                      ↘ 审核不通过 → 审核不通过 → 编辑 → 草稿
                      ↘ 撤回 → 草稿

变更（仅待发布及之前）：
  草稿/审核不通过/待发布 → 变更（覆盖原内容，版本号+1，写版本历史）→ 草稿/待发布 → 提交审核 → ...

补遗（仅已发布后）：
  已发布 → 新建补遗公告（独立记录）→ 草稿 → 提交审核 → 已发布

终止：
  已发布 → 终止 → 已终止
```

**关键规则**：
- **不可变更边界**：公告审核通过且到达发布时间（对外可见）后，不可变更，仅可发布补遗公告。
- **可变更区间**：草稿、审核不通过、待发布（审批通过但未到发布时间）状态均可变更，变更覆盖原公告内容，保留版本历史。
- **待发布状态变更**：待发布状态变更后，回退到草稿状态，需重新提交审核。
- **撤回**：待审核状态可撤回，撤回后回到草稿。
- **终止**：仅已发布状态可终止，终止后公告对外不可见（或标记为已终止），不可恢复。

##### 4.1.4.2 补遗公告状态

| 状态 | 状态码 | 说明 | 允许操作 |
|------|--------|------|---------|
| 草稿 | `DRAFT` | 编制中 | 编辑、提交审核、删除 |
| 待审核 | `PENDING_APPROVAL` | 已提交，审核中 | 查看、撤回 |
| 审核不通过 | `APPROVAL_REJECTED` | 审核拒绝 | 编辑、提交审核、删除 |
| 已发布 | `PUBLISHED` | 已对外发布 | 查看 |

**说明**：补遗公告无「待发布」状态，审批通过后按发布时间自动发布（同原公告逻辑）。补遗公告不支持再变更和再补遗。

#### 4.1.5 业务流程

##### 4.1.5.1 公告发布流程

```
项目经理新建公告 → 填写公告信息 → 保存草稿
    │
    ├── 保存草稿 → 草稿状态，可继续编辑
    │
    └── 提交审核 → 待审核 → 进入任务中心
                        │
                        ├── 审批通过 → 待发布（等待到达发布时间）
                        │                │
                        │                └── 到达发布时间 → 已发布（对外可见）
                        │
                        └── 审批不通过 → 审核不通过 → 可编辑后重新提交
```

##### 4.1.5.2 公告变更流程

```
草稿 / 审核不通过 / 待发布 → 点击变更 → 编辑公告内容
    │
    ├── 保存（版本号+1，写版本历史）→ 回到草稿/待发布
    │
    └── 提交审核 → 待审核 → ...（同发布流程）
```

**变更规则**：
- 变更时保留原版本快照到版本历史子表
- 主表更新为最新版本内容，版本号自增
- 待发布状态变更后回退到草稿，需重新提交审核
- 已发布状态**不可变更**

##### 4.1.5.3 补遗公告流程

```
已发布公告 → 点击「发布补遗」→ 新建补遗公告（独立记录，关联原公告）
    │
    ├── 保存草稿 → 草稿状态
    │
    └── 提交审核 → 待审核 → 审批通过 → 到达发布时间 → 已发布
                                    ↘ 审核不通过 → 可编辑后重新提交
```

**补遗规则**：
- 仅已发布状态的原公告可发布补遗
- 一条原公告可发布多次补遗，补遗序号自增
- 补遗公告拥有独立的状态流转和审批流程
- 补遗公告不可再变更和再补遗

##### 4.1.5.4 公告终止流程

```
已发布 → 点击「终止」→ 二次确认 → 已终止
```

**终止规则**：
- 仅已发布状态可终止
- 终止后公告对外标记为已终止
- 已终止不可恢复
- 终止不影响已发布的补遗公告

#### 4.1.6 级联时间默认值规则

公告的时间体系以**公告开始时间**为基准锚点，区域④标段公告信息中的各时间字段按级联关系自动计算默认值。不同采购方式的默认偏移量不同。

**时间级联关系**（公开招标 / 询比 / 谈判）：

```
公告开始时间（基准，区域③）
    │
    ├─→ 文件获取开始时间（默认 = 公告开始时间，只能选择之后的时间）
    │       │
    │       └─→ 文件获取截止时间（默认 = 公告开始时间 + N天）
    │               │
    │               └─→ 澄清截止时间（默认 = 公告开始时间 + M天，≥ 文件获取截止时间）
    │                       │
    │                       └─→ 截标/开标时间（默认 = 公告开始时间 + K天，≥ 澄清截止时间）
```

**各采购方式默认偏移量**：

| 时间字段 | 公开招标 | 询比/谈判（公开） | 竞价（公开） |
|---------|---------|----------------|------------|
| 文件获取开始时间 | +0天（=公告开始时间） | +0天（=公告开始时间） | 无此字段 |
| 文件获取截止时间 | +5天 | +3天 | 无此字段 |
| 澄清截止时间 | +6天 | +3天 | 无此字段 |
| 截标/开标时间 | +21天 | +5天 | 无此字段 |

**校验规则**：
- 文件获取开始时间 ≥ 公告开始时间
- 文件获取截止时间 ≥ 文件获取开始时间
- 澄清截止时间 ≥ 文件获取截止时间
- 截标/开标时间 ≥ 澄清截止时间
- 以上校验在提交审核时执行，校验失败滚动定位到对应字段

> **注**：竞价（公开）无上述时间字段，标段公告信息使用竞价参数字段（见 §4.1.3.2）。

> **法定公告期说明**：公开招标截标/开标时间默认 +21天，隐式保障了《招标投标法》规定的 ≥20日公告期。询比/谈判/竞价的法定公告期待业务确认（当前按默认偏移量执行，后续支持配置化）。

#### 4.1.8 审批流程

| 业务 | 审批路径 | 说明 |
|------|---------|------|
| 公告发布 | 本级审批 → 二级单位审批 | 审批通过后进入待发布状态 |
| 公告变更 | 本级审批 → 二级单位审批 | 待发布状态变更后需重新审批 |
| 补遗公告发布 | 本级审批 → 二级单位审批 | 独立审批，与原公告审批互不影响 |

**审批通过后发布逻辑**：
- 审批通过后状态变为「待发布」
- 系统定时检查发布时间，到达发布时间后自动将状态变为「已发布」
- 若发布时间已过（早于当前时间），审批通过后立即发布

#### 4.1.9 关键业务规则

1. **一个项目一个公告**：一个采购项目仅可有一个非终止状态的公告，已终止后可新建新公告。
2. **公告类型自动判定**：根据项目采购方式自动判定公告类型，不可手动修改。
3. **6 大区域结构**：公告页面分为项目信息、标段/包信息、公告信息、标段公告信息、采购单位信息、其他相关说明 6 个区域。区域①②③⑤⑥为公告主表（项目级），区域④为标段公告子表（标段包级，页签切换）。
4. **级联时间默认值**：区域④的时间字段以公告开始时间为基准锚点，按采购方式自动计算默认偏移量（招标类 +5/+6/+21天，询比/谈判 +3/+3/+5天），用户可手动修改，修改公告开始时间时未手动修改的字段同步更新。
5. **竞价参数回退标段包**：竞价类公告的区域④使用竞价参数字段，从标段包配置中只读带入，不可在公告中修改。
6. **模板文本按采购方式填充**：区域⑥其他相关说明的系统内置模板按采购方式自动填充（招标类含标书款支付和异议条款，非招标类含平台使用费），用户可编辑覆盖。
7. **公告发布媒体差异**：招标类默认勾选采购官网 + 中国招投标公共服务平台，非招标类仅采购官网。
8. **不可变更边界**：公告审核通过且到达公告开始时间（对外可见）后，不可变更，仅可发布补遗公告或终止。
9. **可变更区间**：草稿、审核不通过、待发布状态可变更，变更覆盖原内容，保留版本历史。
10. **补遗独立性**：补遗公告为独立记录，复用原公告主表结构但无需标段公告信息，拥有独立状态流转和审批流程。
11. **终止不可恢复**：公告终止后不可恢复，已终止不影响已发布的补遗公告。
12. **报名管理不在公告模块**：公告模块仅负责发布公告，供应商报名管理放到标前准备模块。

### 5.2 各模块内部状态

各模块内部状态独立设计（如公告模块的草稿→待审核→待发布→已发布），模块间通过项目执行状态流转衔接。

> 详细的项目执行状态机设计待各模块设计完成后统一整合。

## 六、角色权限映射

| 功能 | 角色 | 权限说明 |
|------|------|---------|
| 公告编制 | `BUSINESS_USER` / `BUSINESS_MANAGER` | 项目经理及项目组成员 |
| 公告审核 | 审批流配置的审批人 | 根据审批流配置 |
| 公告发布 | 系统 | 审批通过后到达发布时间自动发布 |
| 补遗公告编制 | `BUSINESS_USER` / `BUSINESS_MANAGER` | 项目经理及项目组成员 |
| 补遗公告审核 | 审批流配置的审批人 | 根据审批流配置 |

> 委托采购场景下，代理项目经理拥有同等操作权限。其余模块权限待各模块设计时补充。
