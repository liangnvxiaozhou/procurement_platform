# 审批流式记录视觉方案 (按单位分组)

---

## 1. 设计原则

*   **分组显示**：以 **审批单位 (Organization)** 为维度进行卡片式分组展示。不同单位内部节点数量可能不同，通过分组聚合，避免长列表造成的视觉混乱。
*   **层级清晰**：将属于同一组织的所有审批节点收纳在该组织的大卡片内。
*   **纯记录属性**：本方案仅定义视觉展示与字段信息，作为只读的审批轨迹查看（不包含审批操作逻辑）。
*   **状态可视化**：通过颜色、展开/收起状态区分流程当前所在的环节。

---

## 2. 视觉状态规范

根据流程进行到的不同阶段，单位卡片分为三种视觉状态：

| 状态类别 | 视觉标识 | 默认行为 | 说明 |
| :--- | :--- | :--- | :--- |
| **已完成的单位** | ✅ 绿色实心对钩 | **默认收起** | 代表该单位下所有节点均已完成。摘要可显示“全节点已通过”。展开后可查看该单位内部的详细流转历史。 |
| **当前所在的单位** | 🔵 蓝色呼吸/高亮 | **默认展开** | 审批流正在进行中的单位。直观展示该单位内各节点的状态（已完成的子节点 + 当前处理中的节点）。 |
| **未到 达的单位** | ⚪ 灰色虚线/占位 | **默认收起** | 流程尚未流转到的单位。仅展示预设的节点名称信息，人员显示"--"或“由上级指定”。 |

## 4. 字段级展示规范

每个分组内的子项（具体节点记录）包含以下核心信息：

| 字段名 | 展示逻辑与示例 |
| :--- | :--- |
| **审批时间** | **已到达/已完成节点**：显示实际发生的时间（如 `2026-04-17 10:00`）。<br>**未到达节点**：不显示或显示 `--`。 |
| **审批节点** | 显示配置的节点名称（如：`部门经理审批`、`总经理审批`）。 |
| **审批人** | **已处理**：显示姓名（如 `张三`）。<br>**处理中**：显示姓名并加粗高亮（如 `**王五**`），提示用户关注。<br>**待指定**：显示 `--`、`待定` 或 `待上一级指定`。 |
| **意见/状态** | **已通过**：显示 `同意` 及具体的审批意见（如 `同意，预算合理`）。<br>**已驳回**：显示 `驳回` 及驳回原因（如 `资料缺失`）。<br>**待处理**：显示 `等待中`。 |

---

## 5. 状态展示差异与截断逻辑

审批流记录在展示时，应根据流程的最终运行状态（进行中 vs 不通过/终止），采用不同的渲染策略：

### 5.1 场景 A：审批进行中 (Running)
*   **展示策略**：**初始化展示全量审批流**。
    *   即使某些单位及其节点尚未流转到（状态为 PENDING），也必须完整显示在列表中。
*   **设计目的**：**进度预期**。
    *   通过展示完整的预设路径（如：本级 -> 二级 -> 采购中心），让审批人和发起人清晰感知到流程的宏观走向和剩余跨度。
    *   **视觉表现**：已到达节点高亮，未到达节点显示为灰色占位。

### 5.2 场景 B：审批不通过/终止 (Rejected)
*   **展示策略**：**路径截断（截止到当前节点）**。
    *   记录列表应仅渲染至最终导致流程失败的**“不通过节点”**。后续未发生的节点不再渲染或直接在视觉上移除。
*   **设计目的**：**异常聚焦**。
    *   既然业务规则定义“不通过即从头审批”，意味着当前流实例已实质性终结。截断列表能迅速让用户视线聚焦于导致终止的那个红色错误节点，忽略无关的后续预设节点。
    *   **视觉表现**：
        *   **当前不通过节点**：使用 **红色** 高亮，必须醒目展示“驳回原因”。
        *   **后续节点**：移除或隐藏。
