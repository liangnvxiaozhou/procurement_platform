# 产品设计需求文档撰写 Skill

> 基于采购计划功能设计文档撰写过程提炼
> 创建日期：2026-05-07
> 适用场景：企业级后台系统产品需求文档（PRD）撰写、评审、迭代

---

## 一、工作流程

### 1. 理解阶段
- 先完整阅读原始文档，理解整体结构和业务逻辑
- 收集用户的修改想法，逐条确认意图，不假设
- 主动 review 文档，发现结构性和逻辑性问题
- 提出改进建议，让用户选择采纳范围

### 2. 规划阶段
- 形成修改计划表（修改内容、影响章节、优先级）
- 识别全文关联影响（章节编号、交叉引用、命名一致性）
- 确认后再执行，避免返工

### 3. 执行阶段
- 按优先级逐项修改
- 每次修改后更新"最后修改时间"到秒级
- 保持老文档不变，新文档独立版本
- 修改完成后提供变更汇总

### 4. 迭代阶段
- 用户审阅后提出新的修改点
- 批判性分析开发/用户的建议
- 提出混合方案，权衡利弊后确认
- 持续记录待确认问题

## 二、文档结构规范

### 1. 标准章节组织

```
一、功能概述（模块划分、业务定位）
二、核心业务流程（编制→审核→执行→立项）
三、数据结构（按业务实体分组，体现层级关系）
四、菜单结构设计
五、页面设计（列表页、新增页、详情页、变更页）
六、关键业务规则（编制/审核/分派/变更/立项规则）
七、Excel 导入设计
八、审批流程集成
九、角色权限映射
十、状态字典（计划状态/子计划状态/明细状态/立项状态）
十一、待确认问题
```

### 2. 章节层级原则
- 最深不超过三级（如 3.1.3.1）
- 数据结构按业务实体分组，体现层级关系
- 示例：年度计划 > 子计划 > 明细

```
三、数据结构
├── 3.1 年度计划
│   ├── 3.1.1 年度计划结构
│   ├── 3.1.2 年度计划主表字段定义
│   └── 3.1.3 子计划字段定义
│       ├── 3.1.3.1 工程/服务类明细字段
│       └── 3.1.3.2 物资类明细字段
├── 3.2 项目计划
│   ├── 3.2.1 项目计划结构
│   └── 3.2.2 项目计划主表字段定义
└── 3.3 采购项目立项
    └── 3.3.1 立项主表字段定义
```

### 3. 字段定义表规范
- 统一格式：`字段 | 类型 | 必填 | 说明`
- 必填标识：✅ / ❌ / 条件必填 / 系统自动
- 系统字段统一包含：创建人、创建时间、更新人、更新时间
- 状态字段标注参见章节（如"参见 10.1.3 子计划状态"）
- 类型描述准确：文本/下拉选择/级联选择/数字/日期/日期时间/文件上传/文本域/系统自动/自动计算

### 4. 状态设计规范
- 每个业务层级独立定义状态（计划、子计划、明细、立项）
- 状态字典包含：状态名、操作权限、说明
- 配套状态流转图（文本图形化）
- 状态与操作按钮严格联动
- 状态命名用中文（草稿/待审核/已生效/审核不通过/已变更/已使用/已建项）

## 三、交互设计原则

### 1. 页面布局选择

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 数据量大（多级嵌套） | 平铺主表 + 列表 + 弹窗编辑子级 | 避免页面过长，弹窗隔离编辑，方便撤回 |
| 数据量小（单层） | 完全平铺 | 简洁直观，方便撤回编辑 |
| 复杂流程（多步骤依赖） | 向导式 | 引导用户，降低认知负担 |

### 2. 平铺+弹窗模式（推荐用于多层级数据）

```
┌─ 新增年度计划 ──────────────────────────────────────────┐
│                                                         │
│  ▾ 基础信息（默认展开，支持折叠）                         │
│  字段表单...                                             │
│                                                         │
│  ▾ 子计划列表                                            │
│  预算使用情况提示                                        │
│  [ 添加子计划 ]  [ Excel导入 ]  [ Excel导出 ]           │
│  子计划列表（含操作列）                                   │
│                                                         │
│  [ 保存草稿 ]  [ 提交审核 ]                              │
└─────────────────────────────────────────────────────────┘

点击"添加子计划"或"编辑" → 打开全屏弹窗：
┌─ 添加子计划 ────────────────────────────────────────────┐
│  ▾ 基本信息                                              │
│  子计划字段表单...                                       │
│  ▾ 明细列表                                              │
│  [ 添加明细 ]  [ Excel导入 ]                             │
│  明细列表...                                             │
│  [ 取消 ]  [ 保存 ]                                      │
└─────────────────────────────────────────────────────────┘
```

### 3. 列表设计原则
- 用**页签**区分不同数据类型（年度计划/项目计划），而非搜索条件
- 操作按钮根据状态动态显示，在文档中明确列出每种状态对应的操作
- 列表字段与详情页字段不必完全一致，列表展示关键信息
- 列表字段需与交互说明保持一致

### 4. 命名统一原则
- 状态字段用中文（草稿/待审核/已生效/审核不通过/已变更）
- 避免同名不同义（如"计划类型"在不同层级含义不同需区分命名）
- 菜单名称准确无歧义（"下属企业"而非"所属企业"）
- 操作按钮命名一致（"提交审核"而非有时"提交"有时"提交审核"）

## 四、质量检查清单

### 1. 一致性检查
- [ ] 状态命名全文统一
- [ ] 字段定义与页面展示一致
- [ ] 操作按钮与状态字典联动
- [ ] 章节编号与交叉引用同步更新
- [ ] 必填标识前后一致
- [ ] 页面布局图中的列表字段与交互说明一致

### 2. 完整性检查
- [ ] 所有状态都有对应的操作按钮定义
- [ ] 所有页面都有字段定义引用
- [ ] 状态流转图覆盖所有分支（审核通过/不通过/撤回）
- [ ] 系统字段不遗漏（创建人/创建时间/更新人/更新时间）
- [ ] 边界条件有说明
- [ ] 校验规则完整（金额约束、类型一致性等）

### 3. 逻辑性检查
- [ ] 业务流程闭环（编制→审核→执行→立项）
- [ ] 数据层级关系清晰
- [ ] 金额校验规则完整
- [ ] 并发场景有考虑（并发审批、数据锁定）
- [ ] 审核不通过后的处理有明确说明

## 五、风险管理

### 1. 并发审批冲突
**风险场景**：多层级同时变更（如子计划明细变更审核中 + 年度计划变更审核中）

**推荐方案**：互斥锁定
- 子计划明细变更审核期间 → 该子计划所属年度计划不可发起变更
- 年度计划变更审核期间 → 该年度计划下所有子计划不可发起明细变更
- 按钮置灰 + 提示："当前计划正在审核中（审核人：XXX），暂不支持变更"

### 2. 待确认问题管理
- 及时记录不确定的问题，编号管理
- 已确认的问题保留在列表中，标记"已确认：XXX"
- 定期回顾，推动确认进度
- 不强行决定未确认的问题

### 3. 版本管理
- 不改动老文档，新文档独立版本
- 文件名包含日期标识（如 `采购计划功能设计_20260428.md`）
- 每次修改更新最后修改时间到秒级
- 修改完成后提供变更汇总

## 六、协作经验

### 1. 与产品经理协作
- 不假设用户意图，主动提问确认
- 提供选项而非单一方案
- 批判性分析建议，提出改进
- 修改前给预览，确认后再执行
- 主动 review 文档结构和逻辑

### 2. 与开发协作
- 理解开发的技术顾虑
- 从产品角度评估方案合理性
- 提出混合方案平衡各方需求
- 记录待确认问题，不强行决定
- 页面布局选择考虑编辑撤回的便利性

### 3. 文档交付
- 新文档独立版本，不覆盖老文档
- 提供清晰的变更汇总
- ASCII 布局图注意对齐和宽度一致
- 表格列宽统一，边框使用 `┌─┬─┐` 和 `├─┼─┤`

## 七、常见陷阱与避免方法

| 陷阱 | 避免方法 |
|------|---------|
| 向导式页面不利于撤回编辑 | 数据量大时用平铺+弹窗，数据量小时用完全平铺 |
| 列表字段与交互说明不一致 | 布局图完成后对照交互说明逐项检查 |
| 状态字典缺少操作按钮定义 | 每个状态都明确"可编辑""可删除""可撤回"等权限 |
| 章节编号变更后引用未同步 | 修改章节结构后全文搜索引用并更新 |
| 并发审批冲突未考虑 | 多层级数据变更时主动分析并发场景 |
| 同名不同义的字段造成混淆 | 检查全文字段命名，确保语义唯一 |
| 系统字段遗漏 | 每个数据表都统一包含创建人/时间/更新人/时间 |
| 审核不通过后处理不明确 | 状态流转图覆盖所有分支，包括审核不通过和撤回 |

## 八、ASCII 布局图绘制规范

### 1. 边框字符
- 外框：`┌─┐` `│` `└─┘`
- 表格：`┌─┬─┐` `├─┼─┤` `└─┴─┘`
- 分隔线：`───` 或 `──────────`

### 2. 对齐原则
- 整体宽度统一，不参差不齐
- 表单字段每行2个，标签和输入框分行对齐
- 表格列宽一致，内容居中或左对齐
- 各区域之间用空行分隔，增加呼吸感

### 3. 必填标识
- 标签前加 `*` 表示必填
- 非必填不加标识

### 4. 示例

```
┌─ 新增年度计划 ───────────────────────────────────────────────────────┐
│                                                                      │
│  ▾ 基础信息                                                          │
│                                                                      │
│  * 年度计划名称  [                    ]  * 项目性质  [      ▼ ]      │
│  * 预算金额      [            ] 万元    * 计划采购时间  [    ▼ ]      │
│  * 预计结束时间  [    ▼ ]             备注  [                    ]    │
│                                                                      │
│  ▾ 子计划列表                                                        │
│                                                                      │
│  预算使用情况：已用 700 万 / 总额 1000 万  |  剩余 300 万             │
│                                                                      │
│  [ 添加子计划 ]  [ Excel导入 ]  [ Excel导出 ]                        │
│                                                                      │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │ 序号 │ 子计划名称 │ 计划类型 │ 预算金额 │ 操作                    │ │
│  │──────┼───────────┼─────────┼─────────┼────────────────────────│ │
│  │  1   │ XX工程    │ 自行采购 │ 500万   │ 编辑 / 删除              │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  [ 保存草稿 ]  [ 提交审核 ]                                          │
└──────────────────────────────────────────────────────────────────────┘
```

## 九、状态流转图绘制规范

### 1. 符号约定
- `→` 表示状态流转方向
- `↘` 表示分支
- 状态名用中文
- 动作标注在箭头上

### 2. 示例

```
首次编制：
  草稿 → 提交审核 → 待审核 → 审核通过 → 已生效
                      ↘ 审核不通过 → 审核不通过 → 编辑 → 草稿 → 提交审核

年度计划变更流程：
  已生效 → 点击变更 → 已变更 → 编辑主表/子计划 → 提交审核 → 待审核 → 审核通过 → 已生效
                                                              ↘ 审核不通过 → 审核不通过

子计划变更流程：
  已生效 → 年度计划变更时点击变更 → 已变更 → 编辑基础信息 → 随年度计划提交审核 → 审核通过 → 已生效
                                                                         ↘ 审核不通过 → 审核不通过
```
