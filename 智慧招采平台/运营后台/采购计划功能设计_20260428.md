## 一、功能概述

采购计划模块用于对集团的采购计划进行制定和管理，为企业提供及时准确的采购计划和执行路线。

本模块支持企业建立年度采购计划、变更子计划及采购项目立项；支持二级单位审核控股投资企业年度采购计划及分派集中采购计划。

### 模块划分

```
采购计划管理
│
├── 1. 本企业采购计划（编制端）
│   ├── 年度计划：年度计划 → 子计划（自行/集采）→ 明细
│   └── 项目计划：项目计划（仅自行采购）→ 明细
│
├── 2. 所属企业年度计划（查看端）
│   └── 二级单位查看下级单位已生效的计划（年度计划/项目计划）
│
├── 3. 集采计划分派（分派端）
│   └── 二级单位将下级单位的集采子计划分派给执行单位
│
├── 4. 本企业可执行计划（执行端）
│   ├── 本企业审核通过的子计划/项目计划
│   ├── 二级单位分派的集采子计划
│   └── 支持变更明细，重新提交审核
│
├── 5. 电煤计划管理（独立页面）
│   └── 电煤类计划独立管理（因业务特殊性，详见后续专项设计文档）
│
└── 6. 采购项目立项（立项端）
    └── 基于可执行计划中的子计划/项目计划进行立项
    └── 拟采购方式自动带入，可编辑

审核入口：采购系统 → 任务中心（统一处理所有计划审核）
```

## 二、核心业务流程

采购计划模块围绕"编制 → 审核 → 执行 → 立项"的完整业务链路展开，支持年度计划和项目计划两种管理模式，并涵盖集中采购计划的特殊分派场景。

### 2.1 编制流程

```
年度计划：
  创建年度计划主表 → 添加子计划 → 添加明细 → 提交审核

项目计划：
  创建项目计划主表 → 添加明细 → 提交审核
```

### 2.2 审核流程（在任务中心处理）

```
年度计划：
  内部审核 → 逐级审核 → 二级单位审核 → 生效
  审核不通过 → 返回编制端，状态变为审核不通过，可编辑后重新提交

项目计划：
  内部审核 → 生效
  审核不通过 → 返回编制端，状态变为审核不通过，可编辑后重新提交
```

### 2.3 分派流程（仅集采）

```
二级单位查看已生效集采子计划 → 选择执行单位 → 分派
```

### 2.4 执行流程

```
查看可执行计划 → 变更明细（需审核）→ 立项
```

### 2.5 立项流程

```
选择可执行计划中的子计划/项目计划 → 选择明细 → 创建立项

约束：
  - 一条明细仅用一次
  - 所选明细的项目类型须一致
  - 拟采购方式自动带入，可编辑
```

```
年度计划主表（1）
  └── 子计划（N）一对多关系
```

- 1个年度计划 → 多个子计划

#### 新增年度计划（向导式）

**步骤导航**：

```
● 步骤1：基础信息  →  ○ 步骤2：添加子计划  →  ○ 步骤3：添加明细  →  ○ 步骤4：确认提交
```

**步骤1：创建年度计划主表**

字段定义参见 [3.2 年度计划主表字段定义](#32-年度计划主表字段定义)。

**步骤2：添加子计划**

字段定义参见 [3.3 子计划字段定义](#33-子计划字段定义)。

**自动带入规则**：
- 项目类型：年度计划项目性质 = 基建/技改/经营/工程承包类 → 默认"工程"；= 大宗物资类 → 默认"物资"
- 项目性质：自动带入年度计划的项目性质

**校验规则**：
- 子计划金额总和 ≤ 年度计划总金额（强制校验，不允许超额）
- 明细估算金额之和 ≤ 子计划估算金额

**交互说明**：
- 顶部实时显示预算金额使用情况（已用金额/剩余金额）
- 添加子计划时，如果金额超额，实时提示并阻止保存

## 六、关键业务规则

### 6.1 编制规则

| 规则 | 说明 |
|------|------|
| 编制顺序 | 先主表 → 子计划 → 明细 |
| 金额约束1 | 子计划金额总和 ≤ 年度计划总金额（强制校验，不允许超额） |
| 金额约束2 | 明细估算金额之和 ≤ 子计划估算金额（强制校验） |
| 计划类型 | 子计划：自行采购/集中采购；项目计划：仅自行采购 |
| 项目类型 | 工程 / 物资 / 服务 |
| 明细类型 | 工程/服务类 → 项目明细；物资类 → 物资明细 |
| 自动带入 | 子计划的项目类型和项目性质根据年度计划自动带入 |
| 自动计算 | 子计划的计划采购时间 = 明细计划采购时间最早值；拟开工/交货时间 = 明细开工/交货时间最晚值 |
| 物资金额计算 | 物资类明细估算金额 = 采购数量 × 预估单价（转换为万元） |

### 6.2 审核规则

| 规则 | 说明 |
|------|------|
| 审核入口 | 统一在"采购系统 → 任务中心"处理 |
| 年度计划审核路径 | 内部审核 → 逐级审核 → 二级单位审核 |
| 项目计划审核路径 | 内部审核 |
| 自行采购计划审核路径 | 内部审核 |
| 变更审核路径 | 内部审核 |
| 审核不通过处理 | 所有类型计划审核不通过后，状态变为"审核不通过"，可编辑后重新提交 |

### 6.3 分派规则

| 规则 | 说明 |
|------|------|
| 分派范围 | 仅可分派计划类型 = "集中采购" 且已生效的子计划 |
| 执行单位范围 | 仅限当前二级单位下属的单位 |
| 分派后状态 | 分派后，子计划出现在执行单位的"本企业可执行计划"中 |

### 6.4 变更规则

| 规则 | 说明 |
|------|------|
| 变更入口 | 在"本企业可执行计划"中点击"变更" |
| 变更范围 | 仅支持明细的新增/编辑/删除，不修改主表和子计划信息 |
| 原计划不变 | 本企业年度计划中的原始记录不变 |
| 变更审核 | 变更需内部审核通过后方可生效 |
| 变更期间 | 变更审核期间，原已生效的明细继续有效 |
| 状态流转 | 已生效 → 变更 → 已变更 → 提交审核 → 待审核 → 审核通过 → 已生效 |
| 立项限制 | 已变更和待审核状态下，该计划不能用于立项 |

### 6.5 立项规则

| 规则 | 说明 |
|------|------|
| 前置条件 | 必须关联可执行计划中的子计划/项目计划 |
| 关联层级 | 关联到明细级别，只要子计划/项目计划下还有未使用明细，该计划就可选 |
| 明细选择 | 选择子计划/项目计划后，必须在其内部选择具体明细 |
| 项目类型一致性 | 同一立项中，所选明细的项目类型必须一致（提交时校验） |
| 采购方式 | 拟采购方式自动带入（第一个子计划的拟采购方式），可编辑，以立项为准 |
| 立项后标记 | 明细被立项使用后，标记为"已使用"，不可再次用于立项 |
| 估算金额 | 取已选明细的估算金额之和 |
| 变更规则 | 已生效的立项可变更，变更后需重新提交审核 |
| 状态流转 | 草稿 → 提交审核 → 待审核 → 审核通过 → 已生效；审核不通过 → 审核不通过 → 编辑 → 草稿；已生效 → 变更 → 已变更 → 提交审核 → 待审核 |
| 撤回规则 | 待审核状态下可撤回，撤回后状态变为草稿 |
| 删除规则 | 草稿和审核不通过状态下可删除 |

### 8.4 审批回调处理

审批流完成后，系统需回调更新业务表状态：

**审批通过**：
- 首次提交：状态变为"已生效"，可编辑锁定解除
- 明细变更：状态变为"已生效"，更新明细数据，标记为已生效
- 立项首次提交：状态变为"已生效"，关联明细标记为"已使用"
- 立项变更：状态变为"已生效"，更新立项数据

**审批不通过**：
- 首次提交：状态变为"审核不通过"，可编辑锁定解除，记录驳回原因
- 明细变更：状态变为"已变更"，变更失效，恢复原明细，可继续编辑
- 立项首次提交：状态变为"审核不通过"，可编辑或删除
- 立项变更：状态变为"审核不通过"，变更失效，可继续编辑

## 十、状态字典

### 10.1 计划状态

#### 10.1.1 采购计划状态

| 状态 | 可编辑 | 可立项 | 说明 |
|------|--------|--------|------|
| 草稿 | ✅ | ❌ | 编制中 |
| 待审核 | ❌ | ❌ | 已提交审批，等待审核 |
| 已生效 | ❌ | ✅ | 审批通过，可执行，可立项 |
| 审核不通过 | ✅ | ❌ | 审批未通过，可编辑后重新提交 |
| 已变更 | ✅ | ❌ | 已发起变更，可编辑明细，不能立项 |

**状态流转图**：

```
首次编制：
  草稿 → 提交审核 → 待审核 → 审核通过 → 已生效
                      ↘ 审核不通过 → 审核不通过 → 编辑 → 草稿 → 提交审核

变更流程：
  已生效 → 点击变更 → 已变更 → 提交审核 → 待审核 → 审核通过 → 已生效
                                      ↘ 审核不通过 → 已变更
```

#### 10.1.2 采购项目立项状态

| 状态 | 可编辑 | 可删除 | 可撤回 | 说明 |
|------|--------|--------|--------|------|
| 草稿 | ✅ | ✅ | ❌ | 编制中 |
| 待审核 | ❌ | ❌ | ✅ | 已提交审批，可撤回 |
| 已生效 | ❌ | ❌ | ❌ | 审批通过 |
| 审核不通过 | ✅ | ✅ | ❌ | 审批未通过，可编辑后重新提交或删除 |
| 已变更 | ✅ | ❌ | ❌ | 已发起变更，可编辑后提交审核 |
| 已建项 | ❌ | ❌ | ❌ | 已关联到后续正式采购项目，仅可查看 |

**状态流转图**：

```
首次编制：
  草稿 → 提交审核 → 待审核 → 审核通过 → 已生效
                      ↘ 审核不通过 → 审核不通过 → 编辑 → 草稿
                      ↘ 撤回 → 草稿

变更流程：
  已生效 → 点击变更 → 已变更 → 提交审核 → 待审核 → 审核通过 → 已生效
                                      ↘ 审核不通过 → 审核不通过

建项流程：
  已生效 → 被后续采购项目关联使用 → 已建项
```

### 10.2 已使用标记 / 已建项状态

| 标记/状态 | 类型 | 所属层级 | 说明 |
|------|------|------|------|
| 已使用（明细） | 布尔值 | 明细级别 | 标记明细是否已被立项关联使用 |
| 已建项（立项） | 状态 | 立项级别 | 表示该立项已关联到后续正式采购项目 |

**说明**：
- "已使用"是明细级别的布尔标记，明细被立项使用后标记为"是"
- "已建项"是立项级别的终态，表示该立项已进入采购执行阶段
- 进入"已建项"状态后，不可再变更或删除
- 两者不属于同一业务阶段，互不影响
- "已建项"属于采购执行阶段功能，详见后续文档

### 10.6 明细状态

| 状态 | 对应计划状态 | 说明 |
|------|-------------|------|
| 未提交 | 草稿 | 明细编制中，计划尚未提交审核 |
| 待审核 | 待审核 | 计划已提交审核，明细等待审批 |
| 已生效 | 已生效 | 计划审核通过，明细可执行 |
| 已使用 | 已生效 + 已立项标记 | 明细所属计划已被采购项目关联使用 |

**联动规则**：
- 明细状态随所属计划状态自动联动
- 当计划被立项关联后，该计划下所有明细状态变为"已使用"
- 集采分派场景：集采子计划分派给执行单位后，明细仍显示"已生效"，待执行单位立项后变为"已使用"
