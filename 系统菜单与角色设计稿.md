# 系统菜单与角色设计稿

本文档基于当前 `quasar-system` 的数据模型，按照 Scrum 中“先摆出需求，再归纳角色，再落权限”的思路，对系统菜单与角色关系做一版阶段性设计。

本文目标不是一次性把所有权限细节定死，而是先形成一份可评审、可拆分、可逐步实现的角色与菜单基线。

## 1. 设计原则

### 1.1 Scrum 视角下的做法

按照 Scrum 的常见做法，本次不先从“系统里该有哪些角色”出发，而是先把需求摆出来：

1. 谁要用系统
2. 他们想完成什么事情
3. 哪些事情是同一类职责
4. 再把职责收敛成角色
5. 最后把角色映射到菜单与数据范围

这样做的原因是：

- 避免一开始就堆很多角色，后面互相重叠
- 避免把“菜单可见”与“数据范围”混为一谈
- 方便后续按 Sprint 拆分实现

### 1.2 当前系统建模前提

当前 `quasar-system` 已具备以下基础能力：

- `sys_role`：角色定义
- `sys_menu`：菜单定义
- `sys_role_menu`：角色与菜单关系
- `sys_user_role_grant`：用户在组织上下文中的角色授权与数据范围
- `sys_org.admin_uid`：组织管理员

因此本设计遵循以下边界：

- 角色解决“能进入哪些功能菜单”
- 数据范围解决“进入后能看到哪些数据”
- 任职关系解决“用户处于哪个组织/部门上下文”

## 2. 需求列表

以下需求采用用户故事形式表达。

### 2.1 平台治理类需求

1. 作为平台超级管理员，我想维护全平台的组织树、菜单、权限、角色与参数配置，以便统一治理整个平台。
2. 作为平台超级管理员，我想查看关键操作日志，以便审计系统管理行为。
3. 作为平台超级管理员，我想跨组织处理异常授权和组织初始化，以便保障平台正常运行。

### 2.2 一级组织治理类需求

1. 作为一级组织管理员，我想管理本一级组织及其下级组织，以便完成集团到下属组织的管控。
2. 作为一级组织管理员，我想查看并维护本组织体系内的用户、账号、任职与授权，以便完成组织级管理。
3. 作为一级组织管理员，我想配置本组织体系内的审批流、业务资格池和共享规则，以便支撑业务运行。
4. 作为一级组织管理员，我想查看本组织体系内的重要日志，以便做管理追踪。

### 2.3 单个组织治理类需求

1. 作为组织管理员，我想管理本组织基础信息、部门、用户与任职，以便维护组织日常运行。
2. 作为组织管理员，我想为本组织用户分配业务角色和范围，以便控制谁能做什么。
3. 作为组织管理员，我想配置本组织审批流和业务资格池，以便支撑本组织业务协同。

### 2.4 组织人事类需求

1. 作为组织用户管理员，我想维护本组织用户、账号、部门与任职，以便完成人员管理。
2. 作为组织用户管理员，我想维护用户默认组织、默认部门和登录账号状态，以便支持登录和切换组织。

### 2.5 组织授权类需求

1. 作为组织授权管理员，我想维护本组织的角色授权，以便给合适的人分配合适的职责。
2. 作为组织授权管理员，我想配置某个用户获得某个角色时的数据范围，以便限制其可见范围。
3. 作为组织授权管理员，我想维护业务资格池与共享关系，以便业务资源在组织之间协同。

### 2.6 业务管理类需求

1. 作为业务管理员，我想进入业务配置与业务处理页面，以便处理本组织业务。
2. 作为业务管理员，我想查看自己职责范围内的数据，以便开展审核、配置和管理工作。

### 2.7 普通业务使用类需求

1. 作为普通业务用户，我想进入自己的工作台和业务页面，以便完成填报、查看、提交等操作。
2. 作为普通业务用户，我不需要看到系统管理菜单，以便减少误操作和页面噪音。

## 3. 角色推演

根据上面的用户故事，可以把职责逐步收敛为以下角色。

### 3.1 平台级角色

#### `SUPER_ADMIN`

定位：

- 平台全局治理角色
- 负责平台基础模型和系统参数

主要职责：

- 管理全平台组织
- 管理全平台菜单
- 管理全平台权限
- 管理全平台角色
- 管理跨组织授权异常
- 查看平台审计日志

推荐范围：

- `scope_required = false`
- 等价于平台级全量权限

### 3.2 组织治理角色

#### `GROUP_ORG_ADMIN`

定位：

- 一级组织管理员
- 管理某个一级组织及其下级组织

主要职责：

- 管理组织树中的一级组织分支
- 管理本组织体系下的部门、用户、账号、任职
- 管理本组织体系下的角色授权和数据范围
- 管理本组织体系下的审批配置和业务资格共享

推荐范围：

- `scope_required = true`
- `scope_type = ORG`
- `scope_object_type = ORG`
- `include_children = true`

#### `ORG_ADMIN`

定位：

- 单个组织管理员
- 管理本组织的综合事务

主要职责：

- 管理本组织基础信息
- 管理本组织部门、用户、账号、任职
- 管理本组织角色授权与范围
- 管理本组织审批流和资格池

推荐范围：

- `scope_required = true`
- 默认当前组织范围

### 3.3 专项管理角色

#### `ORG_USER_ADMIN`

定位：

- 组织侧人员与账号管理角色

主要职责：

- 维护用户档案
- 维护登录账号
- 维护组织和部门任职
- 维护默认上下文

推荐范围：

- `scope_required = true`
- 仅当前组织或部门范围

#### `ORG_ROLE_ADMIN`

定位：

- 组织侧授权管理角色

主要职责：

- 维护角色授权
- 维护菜单授权结果
- 配置用户在某角色下的数据范围
- 维护业务资格池与共享

推荐范围：

- `scope_required = true`
- 仅当前组织范围

### 3.4 业务角色

#### `BUSINESS_MANAGER`

定位：

- 业务管理岗位

主要职责：

- 进入业务功能
- 处理业务配置
- 查看职责范围内业务数据

推荐范围：

- `scope_required = true`
- 可按 `ORG / DEPT / CUSTOM` 授权

#### `BUSINESS_USER`

定位：

- 普通业务操作用户

主要职责：

- 使用工作台和业务页面
- 完成个人业务操作

推荐范围：

- `scope_required = true`
- 常见为 `SELF` 或 `DEPT`

## 7. 角色与数据范围的对应原则

### 7.1 角色与范围不是一回事

当前系统建议继续坚持以下规则：

- 角色决定“能做什么”
- 菜单决定“能进入哪里”
- 数据范围决定“能看哪些数据”

不要把以下概念混用：

- 不要把组织管理员直接理解成“全量可见”
- 不要把有菜单权限直接理解成“能看全组织数据”
- 不要把任职关系直接当成角色授权

### 7.2 范围配置位置

数据范围应配置在“用户获得某个角色”的授权记录上，也就是 `sys_user_role_grant`。

这意味着：

- `sys_role` 定义角色模板
- `sys_role.scope_required` 定义该角色是否需要范围控制
- `sys_user_role_grant` 定义某个用户在某个组织下获得该角色时的实际范围

例如：

- 张三被授予 `GROUP_ORG_ADMIN`
- 授权组织为 `A集团`
- `scope_type = ORG`
- `scope_object_id = A集团`
- `include_children = true`

则表示：

- 张三在 `A集团` 上下文下是一级组织管理员
- 可管理 `A集团` 及其下级组织的数据

## 8. 第一阶段落地建议

为了便于按 Sprint 推进，建议按以下顺序实施。

### Sprint 1

- 固化角色编码
- 补齐菜单编码
- 完成角色与菜单基础绑定

交付物：

- `sys_role` 初始化数据
- `sys_menu` 初始化数据
- `sys_role_menu` 初始化数据

### Sprint 2

- 完成组织管理员授权界面
- 完成用户角色授权界面
- 完成数据范围授权界面

交付物：

- 角色授权查询与保存接口
- 范围授权查询与保存接口

### Sprint 3

- 业务系统消费 `scope`
- 按 `scope` 过滤实际业务数据

交付物：

- remote 授权查询接口
- 业务系统范围过滤实现

## 9. 当前结论

当前阶段建议先采用以下原则：

- 平台只保留一个真正的平台级角色：`SUPER_ADMIN`
- 一级组织采用 `GROUP_ORG_ADMIN`
- 每个具体组织设置 `ORG_ADMIN`
- 人员管理和授权管理可拆为 `ORG_USER_ADMIN` 与 `ORG_ROLE_ADMIN`
- 业务功能使用 `BUSINESS_MANAGER` 和 `BUSINESS_USER`
- 菜单权限与数据范围分开建模和配置

这版文档适合作为后续评审基础。若后面业务模块继续扩展，可在此基础上继续细化按钮权限、审批权限和跨组织共享规则。
